前置

本文章不属于传统的“教学文章”，而是旨在让你更快、更好地理解 FHQ-Treap 的代码实现，并加深你对代码模板的印象。所以，本文章可能需要搭配另外的“教学文章”食用。

认知上的基础

Q：为什么要强调这些基础？

A：尤其这几条，在代码的理解及复现上帮助很大。

  1. 键值（key）呈小根堆。
  2. 下文中的“信息”（val、id…）在树形被拍扁时（中序遍历）呈递增。
  3. 左（子树）小于右（子树）。
  4. 摒弃线段树的执念，Treap 上不只叶子，各个点都代表一个位置。
  5. 代码上，用（子）树根可代表整个（子）树。
  6. 代码上，树根 x < y < z < …。

要说明的是，2、3 两点虽然讲的是同一件事，但这两种不同的理解方法在代码中都有体现，故分条呈现。

Build

在原序列对于信息有序时，我们可以做到线性建树。事实上，这依赖于“新加点在最右端”这个性质。

关于实现，我们考虑刻画加点过程：原树右链上断一条边，此处插入新加点（满足小根堆），然后断出的子树做新加点的左儿子（保证“最右端”）。于是可以用单调队列维护右链（键值递增）。

关于 push_up 的时机，发现建树过程中，只有右链的形态在改变，于是一个点出单调队列时，它的子树结构就不会变了，此时 push_up 即可。

Split

对于 Split 函数，其内部是递归求解的。个人认为，在学习递归类代码的实现时，独立出“函数的效果”更有利于理解。

我们假设 Split(rt, k, x, y) 是一个已经完成的函数，其效果是将 rt 树按 k（信息）分裂到 x 树与 y 树。

由第 3、6 条基础，rt 的信息大小与 k 比较：小于等于——rt 及左子树归 x；大于——rt 及右子树归 y。“归 x”的实现，就是将 rt 整棵树放到 x 上（x = rt），再删去 x 右儿子，继续分裂 rc 树至 x 右儿子和 y（split(rc, k, rc, y)），“归 y”同。

点出两件事：split(rc, k, rc, y) 中，前一个 rc 指 rc 树还需要分裂，后一个 rc 指 x 的右儿子是空缺的，要分裂至此；“删去 x 右儿子”并未直接写出，因为往下的引用会覆盖掉其原先的右儿子。

Merge

就是 Split 的逆操作。注意，值域（即信息区间）不能交叉。

首先要确保小根堆（这是简单的，哪个小哪个做根），其次由第 2 条基础，不论谁为根，都要有 x 在左，y 在右。
